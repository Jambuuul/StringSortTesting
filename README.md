Сравнение стандартных и строковых алгоритмов сортировки в прямом лексикографическом порядке.

## Алгоритмы


| Алгоритм           | Что делает                                  |
| ------------------ | ------------------------------------------- |
| `quicksort`        | 3-way quicksort, полное сравнение строк     |
| `mergesort`        | стандартный mergesort                       |
| `string_quicksort` | тернарный quicksort по символу на глубине d |
| `string_mergesort` | mergesort с LCP между соседними строками    |
| `msd_radix`        | MSD radix sort                              |
| `msd_radix_cutoff` | MSD radix + string quicksort при size < 74  |


Опорный элемент в quicksort и string quicksort: медиана из трёх. Считаются только посимвольные сравнения. Замеры: 2 прогрева + 5 прогонов, берётся среднее.

## Тестовые данные

Мастер-массив 3000 строк, подмассивы 100..3000 с шагом 100. Алфавит 74 символа (`A-Z`, `a-z`, `0-9`, `!@#%:;^&*()-`), длина строк 10..200.


| Тип              | Описание                                             |
| ---------------- | ---------------------------------------------------- |
| `random`         | случайные строки                                     |
| `reverse_sorted` | обратный порядок                                     |
| `almost_sorted`  | почти отсортировано (~n/50 обменов)                  |
| `prefix_shared`  | группы 20..80 строк с общим префиксом 5..30 символов |


## Анализ результатов

Графики в `results/` (данные из `data/benchmark_results.csv`). На каждый тип данных есть графики времени и сравнений по всем алгоритмам. `comparison_grid_random.png` сравнивает три пары на случайных данных.

### В целом

По сравнениям лучше всех MSD radix: около 7000 при n=3000 на random/reverse/almost и около 52000 на prefix_shared. Quicksort и mergesort где-то посередине.

String mergesort и MSD radix + cutoff на большинстве графиков хуже своих «базовых» версий. Это особенности нашей реализации и порога cutoff, а не то что алгоритмы сами по себе плохие.

По времени картина другая. MSD radix быстрый и по сравнениям, и по времени. Mergesort делает меньше сравнений чем quicksort, но медленнее из-за копирования при слиянии. String mergesort один из самых медленных: рекурсия плюс работа с LCP.

### Quicksort vs String Quicksort

На **random** кривые сравнений почти совпадают (около 141000 при n=3000). У случайных строк общих префиксов мало, string quicksort сравнивает по одному символу, но из-за глубины рекурсии выигрыша почти нет. По времени string quicksort чуть быстрее: одного символа достаточно, полную строку гонять не надо.

На **reverse sorted** quicksort сильно деградирует: около 1085000 сравнений, время в разы больше. String quicksort держится около 151000. Стандартный quicksort сравнивает строки целиком и попадает в плохие разбиения, string quicksort сначала режет по первому символу и порядок входа его не ломает.

На **almost sorted** та же история: quicksort около 1107000, string quicksort около 134000.

На **prefix_shared** string quicksort заметно лучше (267000 против 446000). Строки идут группами с общим префиксом, и алгоритм не перечитывает его заново при каждом сравнении.

Итого: string quicksort помогает там, где есть структура в данных или где обычный quicksort разваливается. На чистом random разницы по сравнениям почти нет.

### Mergesort vs String Mergesort

На **random** mergesort около 46000 сравнений, string mergesort около 79000. LCP при слиянии должен экономить сравнения, но перед каждым merge мы заново считаем LCP между соседними строками в обеих половинах. На случайных данных это не окупается.

На **almost sorted** mergesort около 30000, это лучший результат среди сортировок на сравнениях. Большинство слияний просто пропускается. String mergesort около 80000: там где merge всё же нужен, уходит много сравнений на подготовку LCP.

На **prefix_shared** mergesort около 287000, string mergesort около 623000. Длинные общие префиксы делают каждый пересчёт LCP дорогим, а выигрыш при слиянии его не перекрывает.

На **reverse sorted** mergesort около 32000, string mergesort около 87000.

Итого: на всех типах данных mergesort выигрывает по сравнениям. То что видно на графиках для string mergesort, это overhead реализации, а не предел самого подхода.

### MSD Radix vs MSD Radix + Cutoff

На **random** при n=3000: MSD radix около 7263 сравнений, cutoff около 114682, в 16 раз больше.

Дело в пороге cutoff = 74 (мощность алфавита). После первого radix-прохода 3000 строк попадают в ~75 корзин, в среднем ~40 строк в каждой. 40 < 74, значит каждая корзина уходит в string quicksort. Получается один проход MSD и потом по сути string quicksort на всём массиве. Число сравнений становится близким к string quicksort (около 142000), то есть смльно больше чем у обычной версии.

На **reverse sorted** и **almost sorted** MSD radix даёт те же ~7263 сравнений, линия почти горизонтальная. Cutoff около 106000..117000 на всех типах, потому что он смотрит на размер подмассива, а не на то, как отсортированы строки.

На **prefix_shared** MSD radix около 52000 (больше чем на random, потому что длинные префиксы и больше проходов). Cutoff около 241000. На малых n (100) cutoff и string quicksort близки (около 10000), потому что массив ещё не так сильно дробится.

Итого: cutoff по заданию снижает накладные расходы counting sort на мелких кусках, но по сравнениям проигрывает чистому MSD. На графиках сравнений cutoff одна из худших кривых.

### Сводка (n = 3000, сравнения)


| Алгоритм         | random  | reverse   | almost    | prefix_shared |
| ---------------- | ------- | --------- | --------- | ------------- |
| quicksort        | 141 000 | 1 085 000 | 1 107 000 | 446 000       |
| mergesort        | 46 000  | 32 000    | 30 000    | 287 000       |
| string quicksort | 142 000 | 151 000   | 134 000   | 267 000       |
| string_mergesort | 79 000  | 87 000    | 80 000    | 623 000       |
| msd_radix        | 7 000   | 7 000     | 7 000     | 52 000        |
| msd_radix_cutoff | 115 000 | 106 000   | 117 000   | 241 000       |


**Random**: MSD radix доминирует, адаптированные comparison-сортировки не дают выигрыша.

**Reverse sorted**: quicksort улетает вверх, string quicksort и MSD radix стабильны.

**Almost sorted**: mergesort лучший среди сортировок на сравнениях, quicksort худший.

**Prefix shared**: MSD radix и string quicksort сильно лучше quicksort/mergesort, string mergesort из-за LCP-overhead в хвосте.
